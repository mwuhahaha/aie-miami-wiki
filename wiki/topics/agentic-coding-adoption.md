# Agentic Coding Adoption

## Summary

Agentic coding adoption is the shift from occasional AI-assisted coding to treating coding agents as participants in real engineering delivery. On this page, the topic is anchored by David House's AI Engineer Miami talk, [[2026-04-21-david-house-transforming-programming-mindsets-case-studies-in-agentic-coding-adoption|Transforming Programming Mindsets: Case Studies in Agentic Coding Adoption]], which frames adoption as a change in programming habits, team expectations, and review practice. The useful question is not just whether agents can write code, but how engineers learn to scope tasks for them, inspect their output, and fold generated work into ordinary development without losing ownership.

## Why It Matters

This matters because successful coding-agent use depends on workflow design as much as model capability. Teams need shared norms for delegation, review, testing, and accountability so that agent-generated changes increase throughput without making quality harder to reason about. The connection to [[g2i]] makes the topic especially organizational: adoption is treated as a coaching and enablement problem, where engineers need repeated practice and examples before agents become a durable part of how software work gets planned and shipped.

## Synthesis Pattern

The pattern is a case-study view of adoption. Rather than defining [[coding-agents]] abstractly, the connected talk uses engineering behavior as the evidence: what changes when developers rely on agents for meaningful coding work, what support structures make that sustainable, and what review loops keep quality visible. Read with [[g2i]], the topic points toward adoption programs that combine tools, coaching, and process changes so teams learn when to trust agents, when to constrain them, and how to keep human engineering judgment in the loop.
