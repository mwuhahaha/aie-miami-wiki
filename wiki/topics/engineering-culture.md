# Engineering Culture

## Summary

Engineering culture here means the operating habits that make agentic coding usable inside real teams rather than just impressive in demos. Through David House's talk, the topic centers on how engineers learn to scope work for coding agents, keep review loops tight, and decide what human judgment still owns when tools can generate large changes quickly. The G2i connection adds the workforce angle: adoption depends on senior engineers modeling new planning, verification, and mentoring practices so agentic coding becomes a shared team discipline instead of a private shortcut.

## Why It Matters

Agentic coding changes the speed and shape of software work, but engineering culture determines whether that speed turns into reliable delivery. Teams need norms for what work is safe to delegate, how generated code is reviewed, when to intervene, and how responsibility is assigned after an agent produces a patch. Without those norms, the same tools can increase review burden, hide weak assumptions, and create uneven quality. The linked G2i context makes this especially concrete: organizations adopting agentic coding need senior developers who can teach trust, skepticism, and iteration as everyday engineering behaviors.
